# Session du 16 mai 2026

## ColoriMagiques - Problème déploiement Railway résolu

**Symptôme** : Le container démarre mais crash avec 502 / application failed to respond.

**Cause racine** : `railway.toml` avait `startCommand = [\"node backend/src/server.js\"]` qui **remplace** le `CMD` du Dockerfile au lieu de le compléter. Le CMD du Dockerfile fait `sh -c 'node migrations/run.js && node migrations/seed.js && node server.js'` — la chaîne complète est nécessaire pour créer les tables ET lancer le serveur. Sans elle, `server.js` est lancé sans que les tables soient créées → crash.

**Solution** : Retirer `startCommand` du `railway.toml` pour laisser le `CMD` du Dockerfile s'exécuter entièrement.

## Session du 16 mai 2026 - suite (23:08)

**Problème persistant**: Container crash 502 même après suppression du startCommand.

**Analyse logs Railway** : Build Docker réussi, image pushée, mais AUCUN log runtime. Le container plante au tout premier démarrage, avant même que server.js ne s'exécute.

**Nouvelle cause racine identifiée** : `backend/src/routes/reset.js` avait des contraintes SQL corrompues :
- `'noêl'` au lieu de `'books'` dans `products_theme_check`
- `'véhicles'` au lieu de `'vehicles'` dans `products_theme_check`
- `'Oeufs'` au lieu de `'Oufs'` dans les pdf_filenames

Quand le CMD lance `node migrations/run.js && node migrations/seed.js && node server.js`, le `seed.js` fait un `ALTER TABLE products DROP CONSTRAINT... ADD CONSTRAINT...` avec les mauvaises valeurs → plantage au niveau du seed → le CMD ne lance jamais `server.js` → container mort-né.

**Solution appliquée** :
1. `reset.js` réécrit avec contrainte correcte : `('animals','space','dinosaurs','princesses','nature','vehicles','fantasy','ocean','alphabet','color','forme-geometrique','nursery-prints')` + `books`
2. `Dockerfile` simplifié : CMD = `node backend/src/server.js` seulement (migrations déjà appelées dans server.js via `migrate().then(...)`)
